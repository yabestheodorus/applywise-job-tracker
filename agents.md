# AI / Agents

> **Implemented:** the shared `GroqService` lives at `apps/api/src/common/groq.service.ts` (in the global `CommonModule`), with `extractJson({ system, user, schema })` doing the chat call + retry/backoff + Zod validation (generic over the schema → returns its inferred output type). The new-application path (`POST /applications/extract`) and **CV extraction** (`POST /profile/cv`) are wired (prompts in `applications/prompts.ts` and `profile/prompts.ts`); status-update extraction is not built yet. Requires `GROQ_API_KEY` (optional `GROQ_MODEL`, default `llama-3.3-70b-versatile`); returns a clear 503 until the key is set.

## AI Extraction Prompt Strategy (apps/api — common/groq.service.ts)
- Use Groq's OpenAI-compatible chat completions endpoint (`https://api.groq.com/openai/v1/chat/completions`), model e.g. `llama-3.3-70b-versatile`
- **Two separate prompts** (matching the two flows above), each instructed to return ONLY valid JSON:
  - **New-application extraction** (`POST /applications/extract`): schema `{ company, role, location, workArrangement, employmentType, seniority, industry, jobUrl, summary, requirements[], skills[], salaryMin, salaryMax, deadline }`. No new-vs-update detection — the user already chose "Add application".
    - **`requirements[]` vs `skills[]`** are distinct: `requirements` are the qualification/requirement **bullet phrases** (e.g. "2–5 yrs fullstack experience"); `skills` are the explicit named technologies/tools (e.g. "React", "Node.js"). Only `skills` feeds the matched/gap computation below.
    - `workArrangement` (`REMOTE`/`HYBRID`/`ONSITE`) and `employmentType` (`FULLTIME`/`PARTTIME`/`CONTRACT`/`INTERNSHIP`) must be the exact enum tokens; the Zod draft schema uses `.catch(null)` so an off-vocabulary label degrades to null instead of failing extraction. A past/expired deadline resolves to null.
  - **Status-update extraction** (`POST /applications/:id/status/extract`) — **implemented** (`STATUS_UPDATE_SYSTEM_PROMPT`): the application is already known from the URL, so the prompt context is `{ message, currentStage, stages: [{ id, label }], now }` (current status label + the user's stages + the current ISO date-time, so the model can resolve relative dates like "besok"/"next Monday"). Output schema `{ stageId, newStageLabel, note, confidence, formattedMessage, event }` (`statusSuggestionSchema`) — the model maps the message to an existing stage by `id`, or sets `newStageLabel` if nothing fits (never both); `confidence` is `high`/`medium`/`low` (`.catch('medium')`). **`event`** is `{ title, type (INTERVIEW/ASSESSMENT/DEADLINE/FOLLOWUP/OTHER), scheduledAt ("YYYY-MM-DDTHH:mm") }` **or null** — set ONLY when the message states a concrete date/time (interview slot, assessment deadline, etc.); never invent a schedule. **`formattedMessage`** is the original pasted message reformatted as clean Markdown (short paragraphs, `- ` bullets, optional `**bold**`; signatures/quoted-reply junk stripped; same language; nothing invented) — it is stored on the resulting `StatusEvent.rawText` and shown per-update in the timeline as a 2-line clamped preview (`MessagePreview`) that opens a right-side drawer (`Sheet`) with the full text — rendered via the dependency-free `FormattedText` renderer — plus the event's stage + timestamp header. The prompt explicitly handles Indonesian **or** English messages. The client reviews the suggestion (can override the stage / edit the note / edit or drop the event) before `PATCH /applications/:id/status`, which resolves `statusId` (or creates a stage from `newStageLabel`), atomically updates `Application.statusId` + appends a `StatusEvent`, and — if an `event` is confirmed — creates a `ScheduledEvent` that feeds the **Upcoming** agenda.
  - **Template extraction** (`POST /templates/extract`) — **implemented** (`TEMPLATE_EXTRACTION_SYSTEM_PROMPT`): the prompt context is `{ rawText, existingTemplates: [{ id, question, topic }] }` (a pasted filled-in application form + the user's saved Q&A templates). Output schema `{ items: [{ matchedId, topic, question, answer }] }` (`templateExtractionSchema`) — the model pulls every question→answer pair out of `rawText`, normalises each question to a reusable phrasing, assigns a short 1–2 word Title-Case **`topic`** (reusing an existing template's topic verbatim when it fits, else inventing one, defaulting to "Others" — this drives the colored topic pills + filter on the page), and sets **`matchedId`** to an existing template covering the **same topic** (→ that one is updated, preventing duplicates) or null (→ new). It must only use answers present in `rawText` (never fabricate), skip unanswered questions, and never reuse a `matchedId` or emit two new items on one topic. The service drops any hallucinated `matchedId` (validates against the user's real ids) before returning the review draft; the client edits/includes items, then `POST /templates/apply` creates the new ones + updates the matched ones in one transaction.
- All Groq context (existing stages, application data) is scoped to the authenticated user — never pass another tenant's data into the prompt.
- Validate every JSON response with a Zod schema before saving to DB
- If parsing fails, surface a clear error to the user rather than silently failing
- Groq is free tier but rate-limited — add basic retry/backoff handling, and a clear error message if rate limited

## CV Upload & Extraction (apps/api — profile/ module + common/groq.service.ts)

> **Implemented** as `POST /profile/cv` (`FileInterceptor('file')`, 5 MB limit) → `ProfileService.extractFromCv`. Current cut is **parse-only**: the binary is **not** stored (no Supabase bucket yet); we persist `cvRawText` + `cvFileName` + `cvUploadedAt` + `cvParseStatus`. Extraction runs **synchronously** and returns the draft directly (no PROCESSING/poll step yet). The Storage upload + signed-URL bits below are the planned next step.

- **Upload**: accept `pdf` and `docx` only, enforce a max size (e.g. 5 MB), validate MIME type. *(Planned)* Store the raw file in a **private** Supabase Storage bucket under a per-user path (`cv/{userId}/{filename}`); persist `cvFileUrl` (or just the path — serve via short-lived signed URLs), `cvFileName`, `cvUploadedAt`.
- **Text extraction** (`profile/cv-parser.ts`): parse the file to plain text server-side — `pdf-parse` v2 (`new PDFParse({ data: buffer }).getText()`) for PDF, `mammoth.extractRawText` for DOCX. Save the result to `cvRawText`. If the file has no extractable text (e.g. a scanned/image-only PDF), set `cvParseStatus=FAILED` with a clear message ("Couldn't read text from this CV — try a text-based PDF or DOCX").
- **AI structuring**: send `cvRawText` to Groq with a system prompt that returns ONLY valid JSON matching a defined schema:
  `{ fullName, headline, email, phone, location, summary, links{linkedin,github,portfolio}, skills[], yearsExperience, certifications[], languages[], experiences[{company,title,location,startDate,endDate,isCurrent,description,skillsUsed[]}], education[{institution,degree,fieldOfStudy,startDate,endDate}] }`
- Validate the JSON with a **Zod schema** before returning. The schema is the **shared `updateProfileSchema`/`cvExtractedDraftSchema` from `@repo/api`** (`packages/api/src/profile/profile.schema.ts`) — the extracted draft IS a profile draft, so one tolerant schema (nullish strings, null→[] arrays) validates both the AI output here and the `PATCH /profile` body, and drives the web TanStack Form (Zod = Standard Schema). On parse failure, surface the error — never silently drop data.
- **Review-before-save**: extraction returns a DRAFT (not auto-persisted). The user reviews/edits in the profile form, then confirms — only then is the structured data written (merged into `UserProfile` + `WorkExperience`/`Education` rows). On confirm, set `cvParseStatus=COMPLETED`.
- **Merge strategy**: don't blindly overwrite existing manual edits. On the review screen, show extracted values as suggestions; let the user choose to replace or keep current data (skills are merged + de-duplicated case-insensitively).
- Long CVs may exceed the model context — truncate `cvRawText` to a sane char limit before the call, and reuse the same Groq retry/backoff + rate-limit handling as the application extraction path.

## Skill Matching Logic
- Inputs are the application's `skills[]` (the extracted required skills) vs `UserProfile.skills[]` → `matchedSkills` (intersection) and `gapSkills` (required-but-missing). **Not yet implemented** — until a profile exists both arrays are stored empty; the add-application form shows them read-only as "None yet".
- Simple version (good enough for MVP): Groq compares the application's `skills[]` against `UserProfile.skills[]`, returns matchedSkills, gapSkills, and a short rationale string
- With a CV uploaded, enrich the prompt context with `yearsExperience` and the per-role `skillsUsed` so the rationale can reference actual experience ("3 yrs Next.js at Acme") instead of just a flat skill list
- This is a single Groq API call per application, not a complex ML pipeline — keep it simple and explainable

## Interview Training Session (apps/api — interview/ module + common/groq.service.ts)
Three prompts in `interview/prompts.ts`, all Groq, scoped to one application + the user's profile. See Product.md → "Interview Training Session".
- **`INTERVIEW_QUESTION_GENERATION_SYSTEM_PROMPT`** (JSON, via `extractJson`) — given the job (`role`, `requirements[]`, `skills[]`, `seniority`, `summary`), the candidate (`headline`, `summary`, `skills[]`, `yearsExperience`, real `experiences[]`), the application's `gapSkills[]`, and the user's existing template topics, returns `{ questions: [{ category, question, rationale, talkingPoints[] }] }`. Categories: `BEHAVIORAL | TECHNICAL | ROLE_FIT | COMPANY | GAP | LOGISTICS`. Balanced across categories; at least one GAP per notable gap skill and one LOGISTICS; talking points must be grounded in the candidate's real experience (never invented).
- **`INTERVIEW_COACH_SYSTEM_PROMPT`** (JSON) — given a question + its category + the user's attempt + brief candidate context, grades it on a category-appropriate rubric (STAR completeness for behavioral, correctness/depth for technical, gap-plan for GAP, etc.) and returns `{ score 1–5, feedback, improvedAnswer, keyPoints[] }`. The improved answer stays truthful to the candidate's context; key points are short flashcard cues for active recall.
- **`INTERVIEW_MOCK_SYSTEM_PROMPT(ctx)`** (plain text, via the new `GroqService.chatText`) — a conversational system prompt that turns the model into an interviewer for this role. The client holds the running transcript and POSTs it each turn (`POST /interview/sessions/:id/mock`); the model asks one question at a time, reacts, and digs in with dynamic follow-ups. `chatText` is the non-JSON sibling of `extractJson` (same retry/backoff, no `response_format`), used because a mock reply is free-form prose, not structured data.
- **`INTERVIEW_MOCK_REVIEW_SYSTEM_PROMPT`** (JSON, via `extractJson`) — the end-of-mock debrief: given the job + the full transcript, returns `{ summary, strengths[], improvements[], score 1–5 }` assessing only the candidate's answers (`POST /interview/sessions/:id/mock/review`). Web keeps the mock transcript/debrief in the workspace (not the chat component) so switching tabs doesn't reset it.
- **Readiness score** is computed in the service (not the LLM): each reviewed question contributes its self-rating (or 0.7 if not yet drilled), answered-but-unreviewed contributes 0.3 → averaged to 0–100; all reviewed flips the session to `COMPLETED`.
